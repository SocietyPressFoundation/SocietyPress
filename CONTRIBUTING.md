# Contributing to SocietyPress

Thank you for your interest in SocietyPress. Please read this document in
full before filing an issue or opening a pull request — the project has
a deliberate policy about external contributions that differs from most
open source projects.

## What SocietyPress is, and isn't

SocietyPress is free, open source, GPL-licensed software written and
maintained by one person. The source is public so that:

- Anyone can read exactly what the software does
- Societies can deploy, modify, and self-host without permission or fees
- The platform cannot be acquired and locked behind a paywall
- The community has a permanent right to fork and continue the project

What SocietyPress **is not** is a community-developed project. Design
decisions, feature priorities, and the pace of development are all set
by the maintainer. The goal is a coherent, opinionated product — not a
consensus-built one.

## Pull requests

**Pull requests are not being accepted at this time.** The maintainer will
close incoming PRs without merging them, regardless of quality. This is
not a reflection on you or your work — it's a policy choice about how the
project is run.

If you need a change that the maintainer isn't willing or able to make,
the GPL gives you the right to fork the project and make it yourself.
You're welcome to do so.

## Bug reports are welcome

Well-described bug reports are one of the most valuable things a user can
offer. File them at
[github.com/SocietyPressFoundation/SocietyPress/issues](https://github.com/SocietyPressFoundation/SocietyPress/issues).

A good bug report includes:

- **What you expected to happen**
- **What actually happened**
- **Steps to reproduce** (ideally numbered)
- **Your environment:** SocietyPress version, WordPress version, PHP
  version, hosting provider if relevant
- **Screenshots or logs** if they help

Please **do not** report security issues as public bug reports. Follow
[SECURITY.md](SECURITY.md) for those.

## Feature requests are welcome

Feature requests help the maintainer understand what real societies need.
File them at
[github.com/SocietyPressFoundation/SocietyPress/issues](https://github.com/SocietyPressFoundation/SocietyPress/issues)
using the "Feature request" template.

A good feature request includes:

- **The problem you're trying to solve** (not the specific solution)
- **Who would benefit** and roughly how many societies would use it
- **What you're doing today** to work around its absence
- **Any similar features** in other platforms, for reference

Requests may or may not be acted on. A request that closes with a polite
"no" is not a judgment on the idea — the scope of the project is
deliberately bounded.

## Documentation improvements

Typos, factual errors, and clarity improvements in the documentation are
always appreciated. File them as issues with the "Documentation" label.
As with code, PRs for documentation will also be closed — but the issue
itself is very welcome and will typically be addressed quickly.

## Translations

SocietyPress is internationalized and ships with a `societypress.pot`
file. If you'd like to contribute a translation for a new language,
please open an issue first so we can coordinate. Translation files will
be accepted into the repository directly at the maintainer's discretion.

## Community discussion

For general questions, help using SocietyPress, or discussion that isn't
a specific bug or feature request, please use the support forums at
[getsocietypress.org/community](https://getsocietypress.org/community)
rather than GitHub Issues.

## A note on tone

Please read and follow the [Code of Conduct](CODE_OF_CONDUCT.md). Reports
that are abusive, demanding, or entitled will be closed without discussion,
regardless of technical merit.

## Thanks

The time and care you spend understanding the software well enough to
report real problems with it is itself a contribution. The maintainer is
grateful for it.
